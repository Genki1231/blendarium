## 1. 概要と目的

Blendarium は、Blender 内で使用する複数の自作ツールを一つのアドオンに統合して提供する**ツールプラットフォーム**である。

- 個々のツール(クリーンアップ、命名規則、エクスポート支援など)を共通の基盤(レジストリ、UI シェル、設定管理)の上に載せる
- v1 系は **Blender 内で完結**する
- 将来、外部アプリからヘッドレス(UI なし)でツールを一括実行できるよう、初期設計の段階で logic 層と ui 層を分離しておく

### 1.1 プロダクトの位置づけ

当初「統合プロジェクト管理ツール」として構想したが、要件定義の過程で以下に整理した。

- **Blendarium(本プロダクト)**: Blender 内ツール群のプラットフォーム
- **スコープ外(将来の別プロダクト)**: 外部アプリ、レンダー管理、バッチ管理、アセット管理、プロジェクト管理。外部アプリは将来 Blendarium をバッチ処理で起動するクライアントとなる予定

## 3. スコープ

### 3.1 v1 系で作るもの

1. 基盤システム(ツールレジストリ、共通 UI シェル、設定永続化、ログ、エラー隔離)
2. 内蔵ツール群(§5.2、§5.3 のロードマップに従い段階リリース)

### 3.2 やらないことリスト(v1 系スコープ外)

- 外部アプリ(ランチャー、バッチ実行クライアント)
- レンダーキュー / バッチレンダー管理
- アセットライブラリ管理、プロジェクト / ファイル管理、タスク管理
- チーム共有機能(v1 は個人利用に専念。ただし公開配布は行う)
- ネットワークを介する一切の機能(クラウド同期、テレメトリ、自動アップデート)

## 4. 用語定義

| 用語 | 意味 |
|---|---|
| レジストリ | ツールの発見・登録・解除・有効/無効を一元管理する仕組み(登録簿) |
| logic 層 | `bpy` の UI/コンテキストに依存しない処理本体。ヘッドレス実行可能 |
| ui 層 | パネル・オペレーター等、Blender UI に依存する薄い皮 |
| ヘッドレス実行 | `blender -b`(background)による UI なしの Blender 実行 |
| Extensions 形式 | Blender 4.2+ の公式アドオン形式(`blender_manifest.toml` でメタデータを定義) |
| ドライラン | 実際には変更を加えず、実行した場合の結果だけを表示するモード |

## 5. 機能要件

### 5.1 基盤システム

| ID | 要件 |
|---|---|
| FR-CORE-001 | ツールレジストリ: `tools/` フォルダをスキャンしてツールを発見・登録する。v1 のスキャン対象は内蔵フォルダのみ(将来の外部フォルダ拡張を設計上妨げない) |
| FR-CORE-002 | ツール単位の有効/無効切り替えをユーザーが行える |
| FR-CORE-003 | エラー隔離: 1 つのツールの登録失敗・実行時例外が、基盤および他ツールの動作を妨げない。失敗はログに記録し UI で通知する |
| FR-CORE-004 | 設定永続化: ツールごとの設定を保存・復元する共通機構を提供する(方式は TBD-02) |
| FR-CORE-005 | 共通ログ機構: 各ツールが共通の logger を通じて出力できる(出力先はローカルのみ) |
| FR-CORE-006 | ヘッドレス実行規約: 全ツールは logic 層と ui 層を分離し、logic 層は UI コンテキストなしで呼び出し可能であること(実装ガイドラインとして文書化する) |
| FR-CORE-007 | 破壊的操作(データの削除・変更)を行うツールは、実行前プレビューまたはドライランと確認手順を必ず提供し、Undo に対応する |

### 5.2 v1.0 初期搭載ツール

| ID | ツール | 概要 |
|---|---|---|
| FR-T01 | `data_purge` | 未使用(orphan)データの検出と一括パージ。削除対象の一覧プレビュー → 確認 → 実行。再帰的な孤児化にも対応 |
| FR-T02 | `naming_tools` | 命名規則の定義(プリセット + カスタムルール)、違反の一覧表示、一括リネーム(ドライラン付き) |
| FR-T03 | `scene_stats` | シーン統計(オブジェクト数、ポリゴン数、マテリアル数、テクスチャ解像度等)の集計。UI 表示に加え JSON / Markdown へのローカル書き出しに対応 |

v1.0 の 3 ツールは意図的に異なる型(破壊的一括操作 / 設定永続型検査 / 読み取り専用レポート)を選定しており、基盤システムの検証を兼ねる。

### 5.3 ロードマップ(v1.1 以降)

| バージョン | ツール | 概要 |
|---|---|---|
| v1.1 | `export_profiles` | 書き出し先別プロファイル(Unity / Maya 等: 形式、スケール、軸変換、命名規則、出力先)を定義し、事前チェック → コレクション単位の一括 FBX/glTF 出力を行う旗艦ツール。事前チェックは `naming_tools` / `scene_stats` の logic 層を再利用する |
| v1.2 | `material_merge` | 重複マテリアル(`.001` 系)の検出と統合 |
| v1.2 | `collection_organizer` | コレクション整理支援 |
| v1.3 | `render_presets` | カメラ / レンダー設定プリセットの保存・適用 |
| v1.3 | `node_library` | ノードグループの整理・ライブラリ化(アセットライブラリ連携を含むため最終フェーズ) |

## 6. 非機能要件

| ID | 要件 |
|---|---|
| NFR-001 | 対応バージョン: Blender 4.2 LTS 以降。Extensions 形式(`blender_manifest.toml`)で実装する |
| NFR-002 | プライバシー: 全データはローカル保存。ネットワーク通信は一切行わない(テレメトリ・自動アップデート・外部 API 呼び出しの禁止) |
| NFR-003 | コードおよびコード内コメントは英語で記述する。ドキュメントは日本語(公開時 README は英語 + 日本語) |
| NFR-004 | 大規模シーンでも UI をブロックしない(具体的な目標値は TBD-04) |
| NFR-005 | 対応 OS: Windows を第一対象とし、OS 依存コードを避けて macOS / Linux でも動作する実装とする |

## 7. 制約条件

| ID | 制約 |
|---|---|
| CON-001 | ライセンス: GPL-3.0-or-later を想定(`bpy` 依存アドオンは GPL 派生と見なされるのが通説であり、Blender Extensions での配布には GPL 互換が必須)。最終確認は TBD-05 |
| CON-002 | Blender Extensions ポリシー準拠: 自己完結(self-contained)であること。実行コードの外部ダウンロードや自己アップデート機構を持たないこと |
| CON-003 | ツール搭載方式は「全ツール内蔵」。ユーザーフォルダからの動的読み込みは v1 系では実装しない(レジストリ設計で将来拡張の余地のみ確保) |

## 8. アーキテクチャ方針(概要)

詳細は次フェーズ(詳細設計)で確定する。方針レベルの合意事項は以下。

```
blendarium/
├── blender_manifest.toml   # Extensions metadata (Blender 4.2+)
├── __init__.py             # top-level register / unregister
├── core/                   # bpy-independent: registry, settings, logging
├── tools/                  # one subpackage per tool
│   └── <tool_name>/
│       ├── __init__.py     # tool metadata + register/unregister
│       ├── logic.py        # headless-callable core logic
│       └── ui.py           # panels / operators (bpy-dependent)
└── ui/                     # common shell
```

- 各ツールは共通のインターフェース(メタデータ + register/unregister + logic/ui 分離)に従う
- レジストリはフォルダスキャン方式(FR-CORE-001)

## 10. v1.0 受け入れ基準(完成の定義)

1. Blender 4.2 LTS に Extensions 形式でインストール・有効化・無効化が正常に行える
2. 3 つの初期ツールが共通シェル上に表示され、それぞれの機能が動作する
3. ツールの有効/無効切り替えが機能し、無効化したツールの UI が消える
4. 意図的に壊したダミーツールを混入させても、他ツールと基盤が正常動作する(エラー隔離の実証)
5. `scene_stats` の logic 層が `blender -b <file.blend> --python <script>` 経由で実行でき、JSON をローカル出力できる(ヘッドレス規約の実証)
6. 破壊的操作(`data_purge`、`naming_tools` のリネーム)がプレビュー / ドライラン → 確認 → Undo 可能の流れで動作する
7. ネットワーク通信が一切発生しない
